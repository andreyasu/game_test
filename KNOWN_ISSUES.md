# Известные недостатки и сомнения по решению

## Реализованная функциональность

- Мечник: Сокрушающий удар + перемещение
- Охотник: Стремительный выстрел + Удар из тени + перемещение
- Вся система событий (MapCreated, UnitSpawned, UnitMoved, UnitAttacked, UnitDied, MarchStarted, MarchEnded)

---

## Известные проблемы

### 1. Движение при заблокированной цели

Если клетка-цель перманентно занята другим юнитом, `MARCH_ENDED` никогда не сработает.
Юнит будет пытаться встать рядом с целью, но точная позиция никогда не будет достигнута.
Симуляция завершится по условию «нет действий» (`anyActed == false`), но это не идеально.

**Возможное решение:** считать марш завершённым, когда юнит достиг наилучшей доступной позиции
(т.е. следующий шаг привёл бы ровно в цель, но она занята).

### 2. Недетерминированность выбора цели

RNG инициализируется через `std::random_device`, что полностью недетерминировано.
Это соответствует требованиям задания («мы не ожидаем детерминированного результата»),
но усложняет воспроизведение конкретного сценария при отладке.

**Возможное решение:** добавить флаг командной строки `--seed <N>` для фиксированного сида.

### 3. Спаун на занятой клетке

Команда `SPAWN_*` не проверяет, занята ли клетка. Два юнита окажутся на одной позиции.
Это нарушает инвариант «юнит занимает клетку, не позволяя другим на неё встать».

### 4. Мёртвый юнит всё ещё действует в текущем тике

По условию: «При HP ≤ 0 юнит выполняет свои действия в текущем ходу».
Реализовано через снэпшот юнитов в начале тика — все юниты из снэпшота получают ход,
даже если их HP стало ≤ 0 в течение этого же тика. Это корректно по спецификации,
но выглядит контринтуитивно: мёртвый юнит атакует или движется.

### 5. Расстояние — Чебышев (8-направленная сетка)

В задании явно не указан тип дистанции. Использована метрика Чебышева
(max(|dx|, |dy|)), которая естественна для 8-направленного движения.
Если предполагалась манхэттенская дистанция, логика атак и движения изменится.

### 6. Порядок событий в removeDeadUnits

`UNIT_DIED` эмитируется в порядке создания юнитов, а не в хронологическом порядке гибели
внутри тика. При одновременной гибели нескольких юнитов порядок событий может быть
неинтуитивным.
